Exit criteria: every new report writes a row; you can query SELECT cycles, battery_health FROM scans and get a scatter plot. Phase 2B — Population-fitted curves (Week 3–4) — replaces hardcoded cycleAnchors Step 2B.1 — Fit a battery health curve. Offline notebook (Python, scipy): Pull (cycles, battery_health) pairs grouped by device family. Fit health = 100 * exp(-k * cycles) (exponential decay) per family. Fall back to global fit if a family has <50 samples. Output: battery-curves.json with { "global": {k: 0.00042}, "macbook-air-m2": {k: 0.00031}, ... }. Step 2B.2 — Replace the function in engine.ts: // before: linear interpolation of magic numbers // after: import curves from "./battery-curves.json"; export function expectedBatteryHealth(cycles: number, deviceFamily?: string) { const k = curves[deviceFamily ?? "global"]?.k ?? curves.global.k; return 100 * Math.exp(-k * cycles); } Step 2B.3 — Ship the curves JSON via a server function so you can update the model without an app release. Step 2B.4 — Add modelVersion and confidence fields to ReportResult. UI shows "Curve fit: macbook-air-m2 (n=1,247, R²=0.89)". Exit criteria: expectedBatteryHealth is data-driven; you can re-fit weekly without code changes.
