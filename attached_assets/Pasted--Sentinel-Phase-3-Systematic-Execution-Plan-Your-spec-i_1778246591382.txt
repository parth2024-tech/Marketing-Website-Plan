
# Sentinel Phase 3 — Systematic Execution Plan

Your spec is solid and stack-correct. Below is how I'll actually execute it, grouped into shippable milestones so each one is testable on its own. A few small corrections to the spec are flagged inline — read those first.

## Corrections to the spec before we start

1. **Lovable Cloud must be enabled first.** It's not yet — without it there is no Postgres, no auth, no Lovable Emails. I'll enable it as step 0.
2. **Rate limiting**: the platform doesn't have rate-limit primitives yet, so a Postgres-backed counter will be ad-hoc. I'll implement basic per-IP throttling on `POST /api/reports` only (the abuse surface), skip it elsewhere, and rely on body-size + Zod caps for the rest. Flagging this so you know it's not as bulletproof as the spec implies.
3. **`citext` extension** must be enabled in the migration (`create extension if not exists citext;`).
4. **JWT for `/my-reports`**: rather than rolling a custom HMAC JWT, I'll use a signed opaque session row (`my_reports_sessions(token, email, expires_at)`) + HttpOnly cookie. Same UX, simpler, no key-rotation footgun. If you specifically want JWT, say so.
5. **Cron URL**: pg_cron will call the stable `project--<id>.lovable.app` host, not `*.workers.dev`. The HMAC scheme and verification are unchanged.
6. **Switch `wouter` → TanStack Router progressively only where needed** for `/r/:id` SEO/OG. The rest of the app stays on wouter to avoid a giant rewrite. The OG meta tag and `/api/og/:id` route work fine because wouter still renders inside TanStack Start's SSR shell — but `/r/:id` head tags need to be set via the TanStack `__root` head merge, so I'll add a tiny TanStack route `src/routes/r.$id.tsx` that does SSR head + delegates rendering to the existing wouter `Report` page.

## Milestone 2 — Core report API  (ship: server-stored reports)

Server routes under `src/routes/api/`, all using `supabaseAdmin`, all Zod-validated, all returning typed JSON.

- `POST /api/reports` — 256 KB cap, Zod on `rawJson`, IP throttle (10/min, 100/day) via `rate_limits`, Idempotency-Key dedup. 12-char base32 nanoid w/ unique-violation retry. 16-char `claim_token`. Logs `report_created` event.
- `GET /api/reports/$id` — returns result; never email; 404 on `deleted_at IS NOT NULL`.
- Wire `HealthTest.tsx` to POST to `/api/reports`, persist `claimToken` in `localStorage` under `sentinel_claim_<id>`, keep raw JSON in `sentinel_report_<id>`.
- `Report.tsx` data load: server-first → localStorage fallback → if found in localStorage and 404 from server, opportunistically POST with `legacy: true`.

**Done when:** running Health Test creates a row in `reports`, and `/r/:id` loads on a second device.

## What I will NOT do (and why)

- **Rate limit every route** — platform doesn't yet have first-class primitives. I'm doing it on the one abuse-prone endpoint and stopping there.
- **`@vercel/og` for PNG OG images** — its dependency tree pulls in Node-only modules unfit for Workers. SVG renders identically in all major social previews.
- **Rip out wouter** — large, risky, off-spec. Only the OG-needing route gets a thin TanStack wrapper.
- **JWT signing key management** — replaced with opaque DB session token; identical UX, lower risk.
