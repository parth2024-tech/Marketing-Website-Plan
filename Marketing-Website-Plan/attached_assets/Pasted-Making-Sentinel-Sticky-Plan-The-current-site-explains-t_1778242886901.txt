Making Sentinel Sticky: Plan

The current site explains the product well but is one-way — users read, leave, run a script, and never come back. This plan adds reasons to return weekly, stronger differentiation from OEM apps (SupportAssist / Vantage / HP Support), and interactive moments that make the brand feel alive even before the desktop app ships.

Three pillars: Interactive tools you can't run as a script, community + benchmarks (data only Sentinel has), and a content rhythm (weekly cadence).



1. Interactive tools (return-worthy utilities)

These are web-only — they cannot be replicated by copy-pasting a script, which fixes the "why come back" problem.





/risk-calculator — Pick your laptop model, age, daily usage hours, charging habits, avg temps. Instantly see a predicted failure-risk score, top 3 weak components, and projected lifespan. Shareable result card. Lightweight, deterministic model in the browser.



/model-lookup — Search by laptop model (e.g. "Dell XPS 15 9520") → known failure patterns, common thermals, battery wear curves, what Sentinel watches most closely on this model. Pulled from a curated JSON dataset (start with top 50 models).



/habit-audit — 8-question quiz ("do you charge to 100% overnight?", "laptop on bed?", "fans cleaned in last year?") → personalized damage report + which habits are silently killing the machine. Outputs a "habit score" the user wants to improve.



/health-test (upgrade existing) — Add a paste-back box: user runs the script, pastes the JSON/text output, the page parses it and renders a real Sentinel-style mini-report in-browser. Bridges the script and the app — gives instant value before install.

2. Community + benchmarks (data flywheel)

The one thing OEM apps structurally cannot do: show you how your machine compares to everyone else's.





/benchmarks — "How does your laptop compare?" Aggregated charts: avg battery wear by model/age, thermals distribution, SSD lifespan curves. Filter by brand, model, year. Updates as more users join.



/wall-of-fame (and shame) — Longest-lived laptops, healthiest 5-year-old machines, worst thermal offenders by model. Anonymous, opt-in.



/model/[slug] — Per-model pages auto-generated from waitlist + habit-audit data: "Dell XPS 13 9310 — 412 owners tracked, avg battery wear 18%, top failure: thermal throttling at 2.3 yrs." Great for SEO and recurring visits.

3. Content rhythm (weekly reason to come back)





/insights — A blog/feed of short data-driven posts ("Why M2 SSDs are dying 40% faster in ultrabooks", "The 3 charging habits we see in laptops that last 7+ years"). Target 1 post/week.



/sample-report (upgrade) — Add a "Generate a fresh sample" button that produces a new realistic mock weekly report each visit, so the page isn't static. Subscribe-by-email option for "get a sample report every Monday".



Email digest signup — Beyond waitlist: a "Monday Health Letter" — anonymized findings + one insight + one habit tip. Reuses waitlist infrastructure with a digest_opt_in flag.

4. Anti-OEM positioning (sharper /why and /compare)





Add a "Whose side is your laptop's app on?" section to /why — OEM apps are built by the company that wants to sell you a new laptop. Sentinel has no incentive to hide a problem or recommend replacement.



Expand /compare matrix with rows OEMs lose on: cross-brand support, predictive (not reactive), no upsell pop-ups, data export, habit coaching, no telemetry sold.



New page /manifesto — short, opinionated essay on why laptop diagnostics is broken. Anchors the brand voice.

5. Live brand surface





HealthFeed upgrade — On home page, show a global ticker: "1,284 laptops monitored · 47 predicted failures averted this week · avg battery wear: 23%." Numbers move. Even if seeded for now, it makes the site feel alive.



Status page — /pulse — fleet-wide live stats dashboard. Eye candy + recurring visit driver.

Build order (3 phases)





Phase 1 — sticky tools (highest ROI): Risk Calculator, Habit Audit, /health-test paste-back parser, Manifesto, /why + /compare upgrades.



Phase 2 — content + community shell: Insights feed (MDX or JSON-driven), Model Lookup with curated dataset, /pulse live fleet ticker, sample-report regenerate.



Phase 3 — data flywheel: Benchmarks page, per-model dynamic pages, Monday Health Letter email opt-in (extends existing waitlist API).

Technical notes





All Phase 1 + 2 features are frontend-only (deterministic models in TS, curated JSON datasets) — no backend changes required, no Cloud enable needed yet.



Phase 3 extends the existing Express + Drizzle waitlist schema with habit_audit_submissions and digest_opt_in columns, plus a model_stats materialized view. Cloud not required unless you want auth-gated history.



Charts: reuse existing chart components (recharts).



Quizzes / calculators: pure React state, results encoded in URL for shareable links.



New routes follow existing TanStack Start createFileRoute pattern with per-route head() metadata.

Out of scope (intentionally deferred)





Real device telemetry sync to web (requires the desktop app to ship first).



User accounts / login (not needed for any Phase 1–2 feature).



Payments — /pricing already exists; no Stripe yet until waitlist converts.



Phase 1 alone should make the site feel dramatically less "boring" and give users 3–4 distinct reasons to revisit. Approve to start with Phase 1.
